# Steel Hawks Programming Curriculum (HawkAcademy)

This is the **Steel Hawks (FRC 2601) programming curriculum** — a [Docusaurus](https://docusaurus.io/) site that teaches new programmers everything from Java basics to how we actually write robot code in [`Rebuilt2026`](https://github.com/steelhawks). It's designed to be **self-running**: a rookie can read it alone, with no mentor, and the pages are interactive (runnable code, quizzes, decision tools).

This README is for **contributors** — people setting up the site locally to write or edit lessons. If you just want to *read* the curriculum, go to the published site.

---

## 2. First-time setup

```bash
# 1. Clone the repo (if you haven't already)
git clone <repo-url>
cd HawkAcademy/docs        # the site lives in the docs/ folder

# 2. Install dependencies (one time, ~1-2 min)
npm install               # or: yarn

# 3. Start the live dev server
npm start                 # or: yarn start
```

`npm start` opens `http://localhost:3000/HawkAcademy/` in your browser. **Edits to pages reload live** — save a file and the browser updates. Leave this running while you write.

> Note the `/HawkAcademy` in the URL — that's the configured `baseUrl`. Don't drop it.

## 3. How the project is laid out

```
docs/
├── docs/                         ← all curriculum pages live here
│   ├── intro.md
│   └── Training/
│       ├── _category_.json       ← defines the "Training" section
│       ├── Java Basics/          ← a section (category)
│       │   ├── _category_.json   ← label + sidebar position of the section
│       │   ├── simple-basics.md  ← a lesson page
│       │   └── ...
│       ├── WPILIB Basics/        ← labeled "Robot Code Basics"
│       ├── Robot Systems/        ← labeled "Building a Robot"
│       │   ├── subsystem-introductions.md
│       │   └── Control Theory/   ← a sub-section (nested category)
│       │       ├── _category_.json
│       │       ├── calculus-basics.md
│       │       └── ...
│       └── Programming Reference/
├── src/
│   ├── components/               ← reusable interactive components (see §5)
│   │   ├── Quiz.jsx
│   │   ├── Note.jsx
│   │   ├── Dropdown.jsx          ← exported as SolutionDropdown
│   │   ├── JavaRunner/
│   │   └── ControlMethodChooser/
│   └── css/custom.css            ← global styles
├── static/                       ← images and static assets
├── docusaurus.config.ts          ← site config (math plugins, navbar, etc.)
├── sidebars.ts                   ← sidebar (auto-generated from folders)
└── package.json
```

**Key idea:** the sidebar is **auto-generated from the folder structure** (`sidebars.ts` uses `autogenerated`). So to add a lesson, you just add a markdown file in the right folder — no sidebar editing needed.

## 4. Writing a new lesson page

### a. Create the file
Add a `.md` (or `.mdx`) file inside the section folder it belongs to, e.g. `docs/Training/Java Basics/my-lesson.md`.

> Our `.md` files use JSX components, which Docusaurus supports. You can use `.mdx` too; both work.

### b. Add front matter
Every page starts with front matter. Copy a neighboring page's style:

```markdown
---
sidebar_position: 4
title: My Lesson Title
---
```

- `sidebar_position` — orders the page **within its section** (1 = first).
- `title` — the page title and sidebar label.

### c. Order sections with `_category_.json`
A section's label and position come from its `_category_.json`:

```json
{
    "label": "Building a Robot",
    "position": 7
}
```

Current section order (the `position` values): Java Basics `4` → Robot Code Basics `6` → Building a Robot `7` → Programming Reference `8`. To insert a new section, give it a `position` that slots it where you want and bump the others if needed.

### d. Write the content
Standard Markdown, plus our components (next section). Use fenced code blocks with a language for syntax highlighting:

````markdown
```java
public class Example { }
```
````

## 5. Interactive components

Import what you need at the **top of the page**, right under the front matter:

```jsx
import Quiz from '@site/src/components/Quiz.jsx'
import Note from '@site/src/components/Note.jsx'
import SolutionDropdown from '@site/src/components/Dropdown.jsx'
import JavaRunner from '@site/src/components/JavaRunner'
```

### Note — callouts for new terms / takeaways
```jsx
<Note title="New term: subsystem">
A subsystem is the code for one mechanism on the robot.
</Note>
```

### Quiz — end-of-page knowledge check
Pass an array of questions. `correct` is the **0-based index** of the right option.
```jsx
<Quiz questions={[
  {
    prompt: "What does periodic() do?",
    options: ["Runs once", "Runs every 20 ms", "Never runs", "Runs on a button press"],
    correct: 1,
    explanation: "WPILib calls periodic() every 20 ms while the robot is on."
  }
]} />
```

### JavaRunner — a runnable Java editor (compiles + runs in the browser)
```jsx
<JavaRunner
  starterCode={`public class Main {
    public static void main(String[] args) {
        System.out.println("Hello, Steel Hawks!");
    }
}`}
/>
```
Optionally pass `expectedOutput="..."` to add a **Check Solution** button that compares the program's output.

> **Two hard rules for JavaRunner code** (see §7): the program must be a complete `Main` class, and the code must be **pure ASCII**.

### SolutionDropdown — collapsible hint / answer
```jsx
<SolutionDropdown
  label="View Solution"
  explanation="Loop and accumulate the sum."
  code={`int sum = 0;
for (int n : nums) sum += n;`}
/>
```

### ControlMethodChooser — the interactive decision tree (Control Theory)
```jsx
import ControlMethodChooser from '@site/src/components/ControlMethodChooser'

<ControlMethodChooser />
```

> **Building your own component?** Put it in `src/components/`, keep styling theme-aware (use Docusaurus CSS variables like `var(--ifm-card-background-color)` and `var(--ifm-font-color-base)` — **never hardcode `#fff`/`#000`**, or it'll glare in dark mode), and import it the same way.

## 6. Math with KaTeX

Math is already configured (`remark-math` + `rehype-katex`, with the KaTeX stylesheet in `docusaurus.config.ts`).

- **Inline:** `$e(t) = \text{target} - \text{measurement}$`
- **Block:**
  ```markdown
  $$u(t) = K_p\,e(t) + K_i\!\int_0^t e(\tau)\,d\tau + K_d\,\frac{de(t)}{dt}$$
  ```

Write equations as real KaTeX — **never** as a fenced code block or with ASCII `^`/`*`.

## 7. Gotchas (read before you debug)

These are the things that have actually bitten us:

1. **No math in JSX props.** KaTeX only processes page text and component *children*, not attribute strings. `<Note title="$\frac{d}{dt}$">` renders the literal `$\frac{d}{dt}$`. Put math in the **body**, keep titles/labels/quiz strings plain text.
2. **JavaRunner code must be ASCII-only.** The code runner compiles as US-ASCII. A single em dash (`—`), curly quote (`"`), `≈`, etc. inside `starterCode` causes `unmappable character` compile errors. Use `-`, straight quotes, etc. (Prose and quizzes can use any Unicode — this is only about the compiled code.)
3. **Broken links fail the build.** `onBrokenLinks` is set to `throw`. Don't link to a page that doesn't exist yet — name it in prose instead, or create it first. Use relative links between pages (e.g. `[What is PID](./what-is-pid)`).
4. **Restart after editing `docusaurus.config.ts`.** Config changes (plugins, navbar) need a dev-server restart; page edits don't.
5. **Pre-existing CAD anchor warnings.** The build prints "broken anchors" for the `/cad` pages. Those are not yours — ignore them as long as the build ends with `[SUCCESS]`.

## 8. Always verify before committing

```bash
npm run build      # or: yarn build
```

This is the same check CI/deploy runs. It must end with **`[SUCCESS]`**. The build catches broken links, bad MDX/JSX, and config errors that the dev server sometimes lets slide. **Don't commit a page you haven't built.**

Other handy scripts:
- `npm run serve` — serve the production `build/` locally to spot-check the final output.
- `npm run clear` — clear the Docusaurus cache if the dev server gets into a weird state.
- `npm run typecheck` — TypeScript check (for `.ts`/`.tsx` files).

## 9. Deployment

The site builds to static files and is hosted on GitHub Pages.

```bash
# Using SSH:
USE_SSH=true npm run deploy

# Or with a username:
GIT_USER=<your-github-username> npm run deploy
```

This builds the site and pushes to the `gh-pages` branch. Confirm with a mentor before deploying.

## 10. New-programmer onboarding checklist

Hand this to someone joining the programming subteam:

- [ ] Install **Node 20+**, **Git**, and **VS Code**.
- [ ] Clone the repo and run `cd docs && npm install && npm start`. Confirm the site opens at `localhost:3000/HawkAcademy/`.
- [ ] **Read the curriculum in order**, top to bottom: Java Basics → Robot Code Basics → Building a Robot (Subsystem Introductions, then Control Theory). Do the quizzes and run the JavaRunner examples.
- [ ] Make a tiny test edit to a page, save, and watch it hot-reload — that's your feedback loop.
- [ ] Practice writing: add a throwaway page in a section, give it front matter, drop in a `<Note>` and a `<Quiz>`, and run `npm run build` to confirm it's clean. Delete it after.
- [ ] When you contribute for real: create a branch, write the page, **`npm run build` must pass**, then open a PR.

### Where to write what
- **Concept/lesson** → a new `.md` page in the right `Training/` section.
- **A runnable exercise** → a `<JavaRunner>` on that page.
- **A new reusable widget** → a component in `src/components/` (theme-aware styling).

Welcome to Steel Hawks programming. When in doubt: read a neighboring page and copy its conventions.

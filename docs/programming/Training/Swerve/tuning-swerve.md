# Tuning Swerve Drive

You've learned what swerve drive is, and how we program it. Now it's time to make it actually work well on a real robot. This section is about **tuning** — the process of dialing in the right numbers so the robot drives smoothly, follows paths accurately, and aligns precisely.

We'll cover:
- What Phoenix Tuner X is and how to use it
- What lives in `TunerConstants.java` and why it exists
- The steer PID loop and how to tune it
- The drive feedforward and how to tune it
- The auton (PathPlanner) PID gains
- The heading alignment PID
- Slip current and why it matters

---

## What is Phoenix Tuner X?

**Phoenix Tuner X** is CTRE's desktop application for configuring, monitoring, and tuning any CTRE hardware — TalonFX motors, CANcoders, Pigeon2 gyros, and more. You can think of it as a live dashboard specifically for your motor controllers.

> For a full walkthrough of Phoenix Tuner X's interface, see the **[Phoenix Tuner X guide](https://v6.docs.ctr-electronics.com/en/stable/docs/tuner/index.html)**

For swerve tuning, you'll use it for three main things:

1. **Identifying devices** — every motor and encoder has a CAN ID. Tuner X shows you all the devices on the bus and confirms they're healthy.
2. **Setting encoder offsets** — the most important step before the robot can drive.
3. **Plotting signals vs. setpoints** — watch the steer angle and drive velocity in real time to see overshoot, oscillation, or lag.

<Note title="Connecting to the robot">
To connect Phoenix Tuner X to the robot, make sure you are on the same network as the robot (either via the robot's radio Wi-Fi or a USB tether). Then open Tuner X and it will auto-detect devices on the CAN bus.
</Note>

## TunerConstants.java

`TunerConstants.java` is the **single source of truth** for all hardware numbers related to swerve. It is auto-generated by the Tuner X Swerve Project Generator, but we edit it manually after that. Here is what lives in it and why each value exists.

### CAN IDs

```java
// Front Left
private static final int kFrontLeftDriveMotorId = 1;
private static final int kFrontLeftSteerMotorId = 2;
private static final int kFrontLeftEncoderId    = 3;

// Front Right
private static final int kFrontRightDriveMotorId = 4;
private static final int kFrontRightSteerMotorId = 5;
private static final int kFrontRightEncoderId    = 6;

// Back Left
private static final int kBackLeftDriveMotorId = 7;
private static final int kBackLeftSteerMotorId = 8;
private static final int kBackLeftEncoderId    = 9;

// Back Right
private static final int kBackRightDriveMotorId = 10;
private static final int kBackRightSteerMotorId = 11;
private static final int kBackRightEncoderId    = 12;
```

These must match the IDs you assigned in Phoenix Tuner X. Every device on the CAN bus must have a **unique** ID — duplicates cause unpredictable behavior. Use Tuner X to verify and change IDs before running the robot for the first time.

### Module Positions

```java
private static final Distance kFrontLeftXPos  = Inches.of(11.5);
private static final Distance kFrontLeftYPos  = Inches.of(10.5);

private static final Distance kFrontRightXPos = Inches.of(11.5);
private static final Distance kFrontRightYPos = Inches.of(-10.5);

private static final Distance kBackLeftXPos   = Inches.of(-11.5);
private static final Distance kBackLeftYPos   = Inches.of(10.5);

private static final Distance kBackRightXPos  = Inches.of(-11.5);
private static final Distance kBackRightYPos  = Inches.of(-10.5);
```

These are the X/Y coordinates of each module's center, measured from the **robot's center**. X is forward/backward, Y is left/right. These values feed directly into `SwerveDriveKinematics` — if they're wrong, the robot will crab-walk when trying to rotate in place.

Measure carefully with a ruler or CAD. They don't need to be re-tuned — just measured once accurately.

### Gear Ratios

```java
private static final double kDriveGearRatio = 6.026785714285714;
private static final double kSteerGearRatio = 26;
private static final double kCoupleRatio    = 3.857142857142857;
```

These come from your swerve module's mechanical specs (usually found in the CAD files or the manufacturer's documentation for modules like SDS Mk4i). **Don't guess these** — a wrong gear ratio makes every velocity and position reading incorrect.

- `kDriveGearRatio`: how many motor rotations per one wheel rotation
- `kSteerGearRatio`: how many motor rotations per one full steering rotation  
- `kCoupleRatio`: a correction for the fact that when the steer motor turns the module, it also slightly rotates the drive axle due to mechanical coupling

### Wheel Radius

```java
private static final Distance kWheelRadius = Inches.of(2);
```

Wheel radius converts drive encoder rotations into meters of linear travel. This is used in `SwerveModule.java`:

```java
cachedState.speedMetersPerSecond = inputs.driveVelocityRadPerSec * constants.WheelRadius;
cachedPosition.distanceMeters   = inputs.drivePositionRad       * constants.WheelRadius;
```

Wheels wear down over time. If odometry gradually drifts, re-measuring and updating the wheel radius is one of the first things to check. Even a 1mm error compounds over a full match.

### Motor Inversion

```java
private static final boolean kInvertLeftSide  = false;
private static final boolean kInvertRightSide = true;
```

The left and right sides of a swerve drive are physically mirrored, so one side's motors run "backwards" to drive forward. If you get this wrong the robot will spin instead of drive straight. You'll know immediately when testing.

## Encoder Offsets — The Most Critical Step

```java
private static final Angle kFrontLeftEncoderOffset  = Rotations.of(-0.22705078125);
private static final Angle kFrontRightEncoderOffset = Rotations.of(-0.414306640625);
private static final Angle kBackLeftEncoderOffset   = Rotations.of(0.072265625);
private static final Angle kBackRightEncoderOffset  = Rotations.of(0.02197265625);
```

This is **the most important tuning step for swerve drive**. Here's why it matters:

Each module's CANcoder reports an absolute angle — but "zero" is wherever the magnet happened to be when the encoder was installed, which is essentially random. The `EncoderOffset` compensates for this so that when you command a module to point forward, it actually points forward.

**How to set encoder offsets:**

1. Physically align all four wheels to point **straight forward**. Use a straight edge across the robot to confirm.
2. Open Phoenix Tuner X and read the **absolute position** of each CANcoder.
3. That reading *is* the offset for that module. Enter it as a negative value (since we subtract the offset to zero the reading):
   ```java
   kFrontLeftEncoderOffset = Rotations.of(-<reading_from_tuner_x>);
   ```
4. Redeploy code and confirm each module sits at 0° when pointing forward.

<Note title="Getting offsets right">
If your offsets are even slightly wrong, all four modules will point at slightly wrong angles. The robot will drive diagonally instead of straight, and path following will be very inaccurate. Take the time to do this carefully.
</Note>

## Steer Gains — Tuning the Steering PID

```java
private static final Slot0Configs steerGains = new Slot0Configs()
    .withKP(100).withKI(0).withKD(0.5)
    .withKS(0.1).withKV(3.23).withKA(0)
    .withStaticFeedforwardSign(StaticFeedforwardSignValue.UseClosedLoopSign);
```

The steer motor runs a **position PID + feedforward loop** onboard the TalonFX at ~1 kHz. Its job is to rotate the module to the exact angle that `runSetpoint` commands. Steering must be fast and accurate — if it lags, the module pushes the robot sideways during the turn, and cosine scaling reduces drive speed unnecessarily.

| Gain | Role | What it does in steering |
|---|---|---|
| `kP` | Proportional | Main restoring force — proportional to how far off the angle is. Start here. |
| `kD` | Derivative | Damping — slows the motor as it approaches target, prevents oscillation. |
| `kS` | Static friction | Minimum voltage to actually move the steering gearbox. |
| `kV` | Velocity feedforward | Predicted voltage needed to spin at a given speed during the move. |
| `kI` | Integral | Almost always 0 — steering doesn't have steady-state error that kS can't fix. |

**Tuning process:**

1. Set all gains to 0. Set `kS` to a small value (0.1) so the motor can overcome friction.
2. **Raise kP** until the module snaps to commanded angles sharply. It will start to oscillate (ring) before settling — that's your upper bound.
3. **Add kD** to damp the oscillation. Raise it until the module moves to the target with one clean motion and no ringing.
4. **Set kV** using the formula: `kV ≈ 12V / (max_steer_speed_rot_per_sec)`. For most modules this is around 2–4.
5. Verify in Tuner X: plot `Position/Setpoint` and `Position/Measured` — they should track each other tightly with no oscillation.

Our current values (`kP=100`, `kD=0.5`, `kS=0.1`, `kV=3.23`) are good starting points for Kraken X60 motors on an SDS Mk4i module. If you're tuning a different robot, start lower and work up.

<Note title="Motion Magic for steering">
As you saw in the previous section, we use Motion Magic on the steer motor which adds a velocity/acceleration profile on top of the position PID. This means the steer motor smoothly accelerates and decelerates instead of demanding maximum voltage instantly. The profile limits are set here:
```java
turnConfig.MotionMagic.MotionMagicCruiseVelocity   = 100.0 / constants.SteerMotorGearRatio;
turnConfig.MotionMagic.MotionMagicAcceleration      = cruiseVelocity / 0.100;
```
If steering feels sluggish, raising `MotionMagicCruiseVelocity` speeds it up. If it overshoots, lower it or raise `kD`.
</Note>

## Drive Gains — Tuning the Drive Feedforward

```java
private static final Slot0Configs driveGains = new Slot0Configs()
    .withKP(0.1).withKI(0).withKD(0)
    .withKS(0.22158).withKV(0.81840);
```

The drive motor runs a **velocity PID + feedforward loop**. Its job is to spin the wheel at the exact speed that `runVelocity` commands. Since velocity loops don't tend to oscillate, `kP` is typically small, `kD` stays at 0, and the bulk of the work is done by `kV`.

| Gain | Role |
|---|---|
| `kS` | Overcomes static friction — the minimum voltage needed to get the wheel spinning at all |
| `kV` | Predicts the voltage needed per unit of velocity. Most of the steady-state control comes from here. |
| `kP` | Corrects any remaining error between the measured velocity and the setpoint. Keep small. |

**How to measure kS and kV (characterization):**

The best way to find drive `kS` and `kV` is to run a **feedforward characterization routine** which we have built into the code as a SysId routine:

```java
public Command driveSysIdQuasistatic(SysIdRoutine.Direction direction) { ... }
public Command driveSysIdDynamic(SysIdRoutine.Direction direction) { ... }
```

These slowly ramp voltage while recording wheel velocity. The resulting data is loaded into a tool (SysId or AdvantageScope analysis) to fit a line:

$$V = k_S \cdot \text{sign}(v) + k_V \cdot v$$

- The **y-intercept** of that line is `kS`
- The **slope** is `kV`

Alternatively, you can estimate `kV` directly:

$$k_V = \frac{12\text{V}}{v_{\text{max}}}$$

where `v_max` is the free speed in radians per second from the motor specs.

### Free Speed

```java
public static final LinearVelocity kSpeedAt12Volts = MetersPerSecond.of(5.12);
```

This is the theoretical maximum speed of the robot at 12V. It's used to desaturate wheel speeds in `runVelocity`. If you measure the robot actually tops out at a different speed, update this value. Running too high causes `desaturateWheelSpeeds` to never actually cap anything; too low limits the robot unnecessarily.

## Slip Current — Protecting Traction

```java
private static final Current kSlipCurrent = Amps.of(CurrentLimits.StatorLimit.driveCurrent);
```

The **slip current** is the motor current at which the wheels start to slip on the carpet. If the drive motor demands more torque than the carpet can grip, the wheel spins without moving the robot — wasting energy and grinding the carpet.

In the code, this value is used in two ways:
1. As the `StatorCurrentLimit` on the drive motor — the motor controller will never exceed this current
2. As the cap for `TorqueCurrent.PeakForwardTorqueCurrent` in FOC mode

**How to find slip current:**

1. Press the robot against a solid wall.
2. Command full drive output while watching motor current in Tuner X or AdvantageScope.
3. The current at which the wheels start squealing or spinning on the carpet is your slip current.
4. Set `kSlipCurrent` to roughly 80–90% of that value for safety margin.

A typical value for Kraken X60 on carpet is 60–80A. Going too low hurts acceleration; going too high causes slipping and brownouts.

## Auton PID Gains

```java
public static final LoggedTunableNumber TRANSLATION_KP = new LoggedTunableNumber("Swerve/TranslationkP", 5.0);
public static final LoggedTunableNumber TRANSLATION_KI = new LoggedTunableNumber("Swerve/TranslationkI", 0.0);
public static final LoggedTunableNumber TRANSLATION_KD = new LoggedTunableNumber("Swerve/TranslationkD", 0.1);

public static final LoggedTunableNumber ROTATION_KP = new LoggedTunableNumber("Swerve/RotationkP", 2.5);
public static final LoggedTunableNumber ROTATION_KI = new LoggedTunableNumber("Swerve/RotationkI", 0.0);
public static final LoggedTunableNumber ROTATION_KD = new LoggedTunableNumber("Swerve/RotationkD", 0.1);
```

These PID gains live in `Constants.AutonConstants` and control the **PathPlanner holonomic drive controller** — the outer loop that keeps the robot on its planned path during autonomous.

This is a **robot-level PID** running on the RoboRIO, not on an individual motor. It takes the error between where the robot currently is (from odometry/pose estimator) and where it should be on the path, and outputs a `ChassisSpeeds` correction that gets added to the path's feedforward velocity.

There are two separate controllers:
- **Translation** — corrects X and Y position errors (in meters). If the robot is 0.1m to the left of the path, this produces a rightward velocity correction.
- **Rotation** — corrects heading error (in radians). If the robot is rotated 5° off the path's target heading, this produces a rotational correction.

**Tuning process:**

Since these are set as `LoggedTunableNumber`, you can adjust them live during test mode **without redeploying**:

1. Enable tuning mode by toggling `Toggles.tuningMode` in AdvantageScope or Shuffleboard.
2. Run a straight-line path and watch the translation error in AdvantageScope.
3. **Raise `TRANSLATION_KP`** until the robot tracks the path tightly. If it oscillates, lower it slightly or add `TRANSLATION_KD`.
4. Do the same for rotation with a path that turns.

<Note title="Units matter here">
Translation kP is in (m/s) per meter of error. A value of 5.0 means: for every 1 meter off the path, add 5 m/s of correction. Rotation kP is in (rad/s) per radian of error. These are very different magnitudes, which is why Translation kP (5.0) and Rotation kP (2.5) look different despite having the same physical meaning.
</Note>

## Heading Alignment PID

```java
public static final LoggedTunableNumber ANGLE_KP = new LoggedTunableNumber("Swerve/AnglekP", 1.0);
public static final LoggedTunableNumber ANGLE_KI = new LoggedTunableNumber("Swerve/AnglekI", 0.0);
public static final LoggedTunableNumber ANGLE_KD = new LoggedTunableNumber("Swerve/AnglekD", 0.0);
```

This is a separate `ProfiledPIDController` used for **heading alignment during teleop** — for example, when the driver holds a button to snap the robot to face a target. It is configured with angular velocity and acceleration constraints (a trapezoidal motion profile) so the heading snaps to its target smoothly instead of suddenly jerking:

```java
mAlignController = new ProfiledPIDController(
    ANGLE_KP, ANGLE_KI, ANGLE_KD,
    new TrapezoidProfile.Constraints(
        MAX_ANGULAR_VELOCITY_RADIANS_PER_SECOND,       // 540°/s
        MAX_ANGULAR_ACCELERATION_RADIANS_PER_SECOND_SQUARED // 920°/s²
    ));
mAlignController.enableContinuousInput(-Math.PI, Math.PI);
```

`enableContinuousInput` tells the controller that -π and +π are the same angle (the controller takes the shortest path around the circle), which prevents the robot from spinning a full 360° when it only needed to turn a few degrees.

**Tuning process:**

1. Bind a button that snaps the robot to a fixed angle (e.g., 0°).
2. **Raise `ANGLE_KP`** until the robot snaps to the target heading quickly. Watch for oscillation.
3. If it oscillates, lower kP and/or raise `ANGLE_KD`.
4. Adjust the motion profile constraints (`MAX_ANGULAR_VELOCITY`, `MAX_ANGULAR_ACCELERATION`) to control how fast the snap feels.

The `alignAtGoal()` method uses a `Debouncer` to confirm the heading has been stable within 5° for 0.5 seconds before reporting "done":

```java
public boolean alignAtGoal() {
    double angleDifference = MathUtil.angleModulus(goal - currentHeading);
    return mAlignDebouncer.calculate(Math.abs(angleDifference) <= Units.degreesToRadians(5));
}
```

This is used to gate actions that require the robot to be aimed before proceeding (like shooting).

## Putting It All Together — Tuning Order

When setting up swerve on a new robot, do these steps in order. Skipping ahead causes confusion because each step depends on the last.

```
1. Assign CAN IDs in Phoenix Tuner X
       ↓
2. Set module positions (measure from CAD or physically)
       ↓
3. Set gear ratios and wheel radius (from module specs)
       ↓
4. Set encoder offsets (align wheels forward, read CANcoder values)
       ↓
5. Verify the robot drives straight (check motor inversion)
       ↓
6. Tune steer gains (kP → kD → kS/kV in Tuner X)
       ↓
7. Characterize drive feedforward (kS, kV via SysId or manual ramp)
       ↓
8. Measure slip current (push against wall, watch current)
       ↓
9. Tune auton translation + rotation PID (run PathPlanner paths, watch error)
       ↓
10. Tune heading alignment PID (snap-to-angle button in teleop)
```
